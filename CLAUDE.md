# 포트폴리오 웹사이트 제작 — Claude Code 작업 브리프

> **이 문서의 용도**
> 신입 게임 기획자 포트폴리오 웹사이트를 처음부터 구축하기 위한 작업 지시서입니다.
> **사용법:** 이 파일을 저장소(`00ny.github.io`) 루트에 `CLAUDE.md`라는 이름으로 두거나, Claude Code 첫 프롬프트로 통째로 붙여넣으세요.
> **현재 상태:** 저장소 `00ny.github.io`는 비어 있고, GitHub Pages는 이미 활성화되어 있습니다(옛 과제는 `study-archive`로 이관 완료). 루트에 `index.html`을 넣고 push하면 `https://00ny.github.io`로 게시됩니다.

---

## 1. 목적과 대상

- **목적:** 국내 게임사 신입 기획 직군 지원용 포트폴리오. 지원서·이력서와 함께 제출/링크된다.
- **주 독자:** 게임사 채용 담당자·현업 리드. **정독하지 않고 스캔**한다. 3초 안에 "볼 가치가 있는가"를 판단한다.
- **성공 기준:** (1) 스캔만으로 핵심 역량이 읽힌다 (2) 관심 있는 사람은 깊이까지 파고들 수 있다 (3) 사이트의 정보 구조 자체가 기획 역량의 증거가 된다.

## 2. 제작자 컨텍스트 — 사이트가 이 철학을 "시연"해야 함

이 사이트는 단순 전시장이 아니라 **작업 샘플**이다. 아래 철학이 화면 구조로 드러나야 한다.

- **구조적 설계 지향:** 플레이어가 정보를 읽고 → 리스크를 계산하고 → 전략적 결과를 만들도록 하는 시스템 설계가 강점.
- **문서화 철학:** `Input → Condition → Output` 상태 전이 명세로, 팀원이 독립적으로 판단할 수 있게 만드는 정밀한 문서화.
- **함의:** 각 프로젝트 페이지의 정보 위계를 이 철학 그대로 구성한다 → **문제 정의 → 설계 의도 → 시스템/상태 명세 → 결과·회고** 순.

## 3. 사이트 구조 (파일/폴더)

```
00ny.github.io/
├── index.html            ← 랜딩: 프로젝트 인덱스 + 외부 링크 + 이력서 섹션
├── README.md
├── .gitignore            ← .DS_Store 포함
├── assets/
│   ├── style.css         ← 전 페이지 공유 CSS (단일 소스)
│   └── img/
└── projects/
    ├── create-01.html    ← 창작 게임 1 (타입 A)
    ├── create-02.html    ← 창작 게임 2 (타입 A)
    ├── reverse-01.html   ← 역기획서 1 (타입 B)
    ├── reverse-02.html   ← 역기획서 2 (타입 B)
    └── reverse-03.html   ← 역기획서 3 (타입 B)
```

- **모든 페이지가 단일 `assets/style.css`를 공유**한다. 디자인 수정은 이 한 파일에서만 이뤄지도록 설계할 것(유지보수·일관성의 핵심).
- `index.html`은 반드시 **루트**에 위치.

## 4. 페이지 설계

### 4-1. 공통 정보 위계 (모든 프로젝트 페이지)
1. **상단 스캔 블록** (5초 파악용, 고정 포맷): 장르 / 한 줄 콘셉트 / 담당 역할 / 핵심 기여 3줄
2. **본문 상세:** 문제 정의 → 설계 의도 → 시스템·상태 명세 → 결과·회고
3. **하단 네비:** 이전/다음 프로젝트 + 인덱스로 돌아가기

### 4-2. 페이지 타입 A — 창작 게임 (2개)
- 위 공통 위계를 그대로 적용. 스크린샷/다이어그램 자리 확보.

### 4-3. 페이지 타입 B — 역기획서 (3개)
- **요약본 = 상단 스캔 레이어**로 배치(채용 담당자가 먼저 읽는 층).
- **원본 = 하단 상세**로 배치하되, `<details>/<summary>` 네이티브 접기로 처리(JS 의존성 없이 스캔 흐름 유지).
- 3개 중 2개는 별도 요약본 보유 → 그대로 상단에. 나머지 1개는 원본에서 3~5줄 요약을 뽑아 상단 포맷에 맞춤.

### 4-4. index.html (랜딩)
- **프로젝트 카드 5개**(창작 2 + 역기획서 3). 각 카드: 썸네일 + 제목 + 한 줄 콘셉트 + `[창작]`/`[역기획서]` 태그칩 + 링크.
- **외부 링크 영역:** 네이버 블로그, 노션.
- **하단 이력서/활동 섹션:** 학력·경력·활동 요약(별도 페이지 아님, 앵커로 이동).

## 5. 콘텐츠 인벤토리 (아직 미제공 — 채워야 할 것)

실제 프로젝트 내용은 아직 전달되지 않았다. **Claude Code는 먼저 뼈대와 플레이스홀더를 완성한 뒤, 아래 표의 항목을 사용자에게 요청**한다.

| 프로젝트 | 필요한 정보 |
|---|---|
| 창작 게임 ×2 | 제목, 장르, 한 줄 콘셉트, 담당 역할, 핵심 기여 3줄, 설계 의도, 주요 시스템/상태 명세, 결과·회고, 이미지 |
| 역기획서 ×3 | 대상 게임, 요약본(2개 보유·1개는 신규 요약), 원본 전문, 분석 관점, 핵심 인사이트 |
| 공통 | 네이버 블로그 URL, 노션 URL, 이력서/활동 텍스트 |

- 플레이스홀더는 `{{제목}}` 처럼 **눈에 띄게 표시**해 나중에 무엇을 채울지 명확히 할 것.

## 6. 비주얼/디자인 방향

- **톤:** 미니멀·가독성 중심의 프로 톤. 화려함이 아니라 **일관성과 정보 위계**로 완성도를 낸다.
- **색:** 거의 무채색 + 절제된 액센트 1색.
- **타이포:** 한글 본문 가독성 최우선. 시스템 폰트 스택 + 한글 폴백(예: Pretendard). 명확한 타입 스케일.
- **레이아웃:** 본문 판독 폭 720~800px, 넉넉한 여백. **모바일 반응형 필수**(채용 담당자가 모바일로 여는 경우 대비).
- **접근성 기본:** 시맨틱 태그, 명도 대비, 키보드 포커스.

## 7. 기술 제약

- **순수 정적 사이트**(HTML/CSS 중심, JS는 최소·선택). 빌드 도구·프레임워크·번들러 **사용 안 함**.
- 외부 의존성 최소화(폰트 정도만 CDN 허용). GitHub Pages에서 그대로 서빙 가능해야 함.
- 최신 브라우저 + 모바일 사파리/크롬 호환.

## 8. 이번 작업 범위 (Claude Code가 지금 할 일)

**목표: 내용만 채우면 되는 "완성된 뼈대"를 만든다.**

1. 위 3장 파일/폴더 구조 생성
2. `assets/style.css` — 디자인 토큰(색·타입·간격) + 공통 컴포넌트(카드, 태그칩, 스캔 블록, `<details>` 스타일, 반응형)
3. `index.html` — 카드 5개(플레이스홀더) + 외부 링크 영역 + 이력서 섹션
4. **타입 A 템플릿 1개**(`create-01.html`) 완성 → 나머지 창작 페이지는 이걸 복제해 채운다
5. **타입 B 템플릿 1개**(`reverse-01.html`) 완성 → 요약본 상단 / 원본 하단(`<details>`) 2단 구조 포함
6. `README.md` + `.gitignore`(`.DS_Store`)
7. 로컬에서 열어 레이아웃·반응형 확인 후, 무엇을 채워야 하는지 **플레이스홀더 목록을 사용자에게 보고**

## 9. 완료 후 배포 절차

```
git add -A
git commit -m "build: 포트폴리오 초기 구조"
git push origin main
```
- push 후 1~2분 뒤 `https://00ny.github.io` 반영.
- GitHub Pages 설정은 이미 `main`/`root`로 되어 있어 추가 설정 불필요(안 되면 Settings → Pages에서 확인).

## 10. 작업 원칙 (요약)

- **일관성 > 화려함.** 페이지 톤이 흔들리지 않게 공유 CSS로 통제.
- 정보 위계를 제작자의 기획 철학(스캔 → 깊이, Input→Condition→Output)에 맞춘다.
- 실제 콘텐츠는 **추측하지 말고** 사용자에게 요청해 채운다.
- 유지보수를 고려해 페이지는 복제 가능한 템플릿 형태로 만든다.
