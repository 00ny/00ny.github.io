## 2. 이번에 내린 판단과 근거 (되돌리기 전에 읽을 것)

### 2-1. "최신 세대"의 정의

토큰 스키마로 세대가 갈렸다. 측정 결과:

- **최신 = 역기획 3종**(헤스티·엔드필드·가마솥). `--font-title/body/mono`, `--slate`, `--observed`, `--patch-soft`, `--maxw` 스키마가 완전 동일. 색 역할까지 주석으로 명세.
- **중간 = 원쁠원.** 팔레트 값은 최신과 동일한데 변수명만 구식(`--disp/--body/--mono`).
- **구세대 = 여우숲.** 다크 테마(#080b11) + Noto 폰트. Pretendard·Hahmlet 미사용.

→ 여우숲은 **가장 오래된** 페이지였다. 최신 3종의 시스템으로 끌어올렸다.

### 2-2. 여우숲 — 히어로의 어둠은 살렸다

최신 세대는 **어두운 히어로(slate) → 밝은 본문 → 어두운 푸터** 패턴이다.
여우숲의 다크 히어로는 이 패턴과 충돌하지 않고 오히려 부합하므로, 히어로의 밤 연출·빛줄기는 유지하고 본문만 라이트로 전환했다.
"통일"은 다크를 전부 지우는 게 아니라 같은 시스템 문법을 쓰는 것이다.

### 2-3. 여우숲 accent = `#4A6B2A` 모스 그린

원래 앰버 `#F2B450`은 라이트 배경에서 **대비 1.71:1로 사용 불가**. 대비를 확보한 앰버(`#96661C`)는 가마솥 앰버와 **색상환 7° 차이**로 충돌.
색상환 빈 구역은 황록(60~110°)과 보라(240~280°)뿐이고 빨강·파랑은 시스템 예약색(patch/link)이다.
→ 모스 그린 선택. 게임 이름 그대로 '숲'에 묶이고, 원쁠원 켈리그린과 46° 이격(기존 가마솥↔헤스티는 2° 차이로 이미 붙어 있음), 대비 5.70:1.

### 2-4. 레거시 별칭을 남긴 이유

여우숲·원쁠원은 구세대 변수명이 **본문 인라인 `style`**에 박혀 있었다(여우숲 15곳). 본문을 건드리지 않기로 했으므로 `:root`에서 별칭으로 연결했다.
`--amber2:var(--accent)` / `--teal:var(--observed)` / `--mono:var(--font-mono)` 등.
**이 줄을 지우면 해당 요소 색이 깨진다.** 자세한 내용은 README의 "레거시 별칭".

### 2-5. 셸은 페이지 토큰을 상속한다

`assets/style.css`의 셸은 색·폭을 하드코딩하지 않고 `var(--accent)`, `var(--maxw)`를 쓴다.
→ 같은 마크업이 페이지마다 그 페이지의 색과 폭에 자동 정렬된다. 원쁠원(880px)과 나머지(1120px)가 각각 맞는 이유.
**셸에 색을 하드코딩하지 말 것.** 이 성질이 깨진다.

### 2-6. 캐러셀 구현 중 지시서(REDESIGN.md)와 달라진 2건

**① 가마솥 캐러셀 이미지 — 전용 크롭을 만들지 않고 `01.webp`를 그대로 썼다.**
지시서는 "01.webp는 세로 슬라이스 소스폭 253px로 부적합 → 02.webp의 기계를 크롭하라"였다. 실제로 해보니:
- 02.webp의 기계는 **머리·앞다리가 피라미드 이미지 블록(x 850~1065)과 겹쳐** 있어, 피라미드를 피해 자르면 머리 없는 기계가 된다.
- hzd 이미지 41장 중 게임 화면은 01뿐이고 나머지는 전부 문서 페이지(1500×1060 표·다이어그램)다.
- 01.webp를 레티나 표시 크기(544px)로 확대해 눈으로 확인한 결과, 삼각 구조물·푸른 게이트·하단의 에일로이가 살아 있고
  소프트함은 대형 구조물의 심도처럼 읽힌다. 지시서가 이미 허용한 여우숲(316px)과 같은 급이다.
→ **01.webp 채택.** `#projects` 카드와 같은 이미지라 인지도 일관된다.

**② 여우숲 캐러셀 이미지 — 지시서에 없던 크롭을 새로 만들었다(`foxforest/carousel.jpg`).**
지시서는 `01.jpg` + `object-position:center 55%`였는데, **그 55%는 아무 효과가 없다**(README 함정 2).
그 결과 타이틀 화면 전체 높이가 나와 **게임 메뉴 버튼이 카드 하단에 잘려** 고장처럼 보였다.
→ 좌측(밤하늘·나무 실루엣·빛나는 두 눈)을 2:3으로 잘라 `carousel.jpg` 생성. 공포 게임 카드로 더 적합하고 8KB로 가볍다.

그 외 구현 중 고친 것: 모바일 중앙 카드의 제목 중복(큰 타이틀과 카드 메타가 겹침), no-JS 폴백의 가로 스크롤,
터치 스와이프 신뢰성(`touch-action:pan-y` + `pointercancel` 처리).

### 2-7. 캐러셀 2차 보완 (2026-07-16, 사용자 지시)

- **카테고리를 제목급으로 승격.** `data-chip` 통합칩(11.5px) → `data-kind`+`data-field` 2개 칩(13px).
  성격(창작/역기획/제안)은 **흰 채움**, 분야(Unity 3D / MSW / 퀘스트·레벨·캐릭터)는 외곽선. 제목 바로 위.
- **설명이 안 보이던 문제 해결.** `#C4CCD4` 14.5px → **흰색 16.5px + 강한 그림자**.
  근본 원인은 색이 아니라 배경이었다 — 원쁠원의 밝은 노을 카드 위에서 회색이 묻혔다.
  → `.cr-title::before` **스크림**(라디얼 그라디언트)을 깔아 카드 밝기와 무관하게 읽히게 했다. **지우지 말 것.**
- **영역 분리 강화.** `.group` 하단 1px 연한 선 → **상단 2px `--ink` 굵은 선**, 여백 52→88px,
  제목 20px → `clamp(25px,3.2vw,32px)`, 편수는 알약 배지로.
- **순서 변경: 원쁠원을 여우숲 앞으로.** 순서는 3곳(캐러셀·카드·셸 체인)에 중복돼 있어 전부 고쳤다.
  셸은 `scratchpad/inject_shell.py`의 `PROJECTS` 배열을 고쳐 재주입했다(재실행 안전). 사이트 바 `01/05` 번호도 따라 바뀐다.

### 2-8. 헤스티 페이지 교체 (2026-07-17, 사용자 지시)

사용자가 헤스티를 새로 작업한 원본 `hesti_operator_proposal_baehyun.html`을 주며 옛 `hesti-character.html`을 대체·삭제하라 지시.
새 파일은 내용(제목·h1·이미지 33장 전부)이 바뀐 **base64 인라인 원본**이라, 5개와 같은 파이프라인을 태웠다:

- **이미지 33개 추출** → `assets/img/hesti-character/`에 덮어씀(옛 번호 이미지 먼저 삭제해 고아 방지). 새 이미지는 옛것과 0개 일치.
- **토큰 최신 정렬(원쁠원식).** 팔레트는 이미 최신인데 변수명만 구식(`--serif/--sans/--mono`, `--risk`, `--obs/--inf/--des`).
  최신 스키마를 추가하고 구식명은 레거시 별칭으로 연결(`--serif:var(--font-title)` 등). `--risk:var(--patch)`.
  **페이지 고유 3색 체계(--obs/--inf/--des = 관찰/추론/설계)와 --sage(히어로 밴드)는 고유값 유지** — 최신 --observed와 값이 다르다.
- **Pretendard Variable → static**(head link + --sans 별칭). style.css 링크 추가.
- **셸 주입**은 `inject_shell.py`의 5번째 슬러그를 새 파일명으로 바꿔 전체 재주입. 이때 **무릉의 '다음' 링크도 자동으로 새 파일명**이 된다.
- **캐러셀/썸네일 재크롭.** 새 히어로 캐릭터(01.webp, sage 배경 전신)로 `carousel.jpg`(2:3)·`thumb.jpg`(16:9) 재생성. sage 배경이라 캐러셀 배경 에코와 색이 맞는다.
- **index 참조 2곳**(캐러셀 카드·프로젝트 카드 href)을 새 파일명으로. 이미지 경로는 `hesti-character/` 유지라 그대로.

**파일명:** 사용자가 준 파일명은 `hesti_operator_proposal_baehyun.html`이었으나, 나머지 4개와 통일해 **`hesti-character.html`로 rename**(사용자 확인).
rename + 참조 3곳(index 캐러셀·카드, 무릉 next) 치환으로 처리. 이미지 폴더는 원래부터 `hesti-character/`라 무변경.

### 2-9. 이력서 채움 (2026-07-17, 사용자 지시)

사용자가 노션 포트폴리오 링크(`glowing-jackfruit-444.notion.site/Hyun-Moment-...` = 추가 링크의 buly.kr가 리다이렉트하는 곳)를 주며 참조를 요청.
Notion은 JS 렌더라 WebFetch로는 헤더만 나온다 — **Browser 도구로 열어 `get_page_text`**로 본문을 뽑았다(스크롤해야 하단 블록 로드됨).

- **대표 항목 선별**(사용자 선택). 노션엔 활동 15개+·스킬 20개+가 있으나 랜딩 이력서엔 과하므로 각 칸 핵심만.
- **연락처: 전화번호 제거, 이메일+SNS만**(사용자 선택). 노션 Contact에도 전화번호가 없다.
  SNS URL은 노션 `a[href]`에서 추출: 유튜브 `@videowoon`, 인스타 `daewae_hwaldong`, 블로그 `cl0udwoon`.
- 채운 4칸: 학력(숭실대·서울고) / 경력·교육(넥슨 메커톤·스마일게이트 스토브크루·NC AI Varco3D·겜마루 기획부장·자격증) /
  활동·수상(펄어비스 등 4개 수상·졸업전시 여우숲·유니데브·장학 3건) / 기술(Figma·Notion·Unity·MSW·Canva·CapCut·Midjourney 등 5카테고리).
- `lede`의 "{{...}} 안내" 문구는 "전체 이력은 추가 링크의 노션 포트폴리오에" 로 교체. buly.kr가 그 노션으로 리다이렉트됨을 curl로 확인.

**노션 참조 방법 메모:** 공개 Notion은 정적 HTML에 본문이 없다. `preview_start {url}` → `get_page_text`가 정답.
`javascript_tool`로 스크롤 시 무거워 타임아웃날 수 있다 — 그래도 `get_page_text`는 이미 로드된 블록을 가져온다.

### 캐러셀 개편 검증 (2026-07-16)

- 순환 5장 무한 루프(5번 넘기면 제자리) · 타이틀·칩·태그라인·배경 에코 동시 전환 통과
- 측면 카드 클릭 = 중앙 이동만(preventDefault) / 중앙 카드 클릭 = 실제 페이지 이동 확인
- hover: 측면 카드 펴짐 + 메타 페이드인 **육안 확인** (요구사항 1 충족)
- 키보드 ←/→ 순환, Tab은 중앙 카드에만 도달(측면 tabIndex=-1)
- 드래그: 좌→다음 / 우→이전 / 40px 미만 무시 (실제 PointerEvent로 검증)
- 목차 앵커 5개 전부 목차 바(52px) 아래 16px 여백에 착지

**도구 한계 메모(다음 세션이 헛수고 안 하도록):** 브라우저 페인의 `left_click`은 해시 앵커 이동을 일으키지 않고,

`resize_window` 후 스크린샷이 축소돼 깨지면 프리뷰를 재시작하고 폭을 다시 지정하면 복구된다.

## 4. 원칙 (유지)

- 일관성 > 화려함. 공통 요소는 `assets/style.css` 단일 소스로만 통제.
- 정보 위계 = 스캔 → 깊이 (Input→Condition→Output 철학의 시연).
- 실제 콘텐츠는 추측하지 않고 사용자에게 요청.
- 프로젝트 페이지 본문은 **의미 변경 금지**. 구조적 주입(셸)과 기계적 치환(경로·토큰명)만 허용.
